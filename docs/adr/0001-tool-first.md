# ADR-0001: Tool-First アーキテクチャの採用

## ステータス

採用済み (2024-01-01)

## コンテキスト

AI エージェントがビジネス操作を実行する際、エージェントに直接ビジネスロジックを実装するか、Tool として抽象化するかを決定する必要がある。

## 決定

**すべてのビジネス操作を Tool として実装する (Tool-First)**

エージェント本体はオーケストレーション（何をするか決める）に専念し、実際の操作は Tool が担う。

## 理由

1. **テスト容易性**: Tool は単体でテスト可能
2. **再利用性**: 同じ Tool を複数のエージェントから使用できる
3. **型安全性**: Tool の引数・戻り値が明示的に定義される
4. **監査**: Tool 呼び出しをログに記録することで操作履歴が追跡可能
5. **LLM 独立性**: Tool の実装を変えてもエージェントのロジックは変わらない

## 結果

- すべての外部システム操作（OpenMetadata, Quarkus API, Kafka）は Tool として実装する
- Tool は `tools/` ディレクトリに配置する
- エージェントは Tool を `create_react_agent(tools=[...])` で受け取る
- Tool 呼び出しはすべてログに記録する

## トレードオフ

| 利点 | 欠点 |
|---|---|
| 高い再利用性 | Tool 数が増えると管理が複雑化 |
| テスト容易性 | Tool 設計が甘いとエージェントが混乱する |
| 操作の可視性 | Tool 境界の定義に時間がかかる |
